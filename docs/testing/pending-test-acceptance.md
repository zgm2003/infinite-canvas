# 待测验收手册

本文把 `docs/pending-test.md` 里的待测项转成可执行验收步骤。它不是自动化测试脚本，也不是正式功能说明。

## 使用规则

1. 先读 `docs/pending-test.md`，确认待测项 ID 和状态。
2. 按本文的场景执行验证，只记录真实观察到的结果。
3. 通过：把对应条目从 `docs/pending-test.md` 移出，再按需要更新 `docs/features.md` 或 `CHANGELOG.md`。
4. 失败：把对应条目标记为 `已测失败`，补充失败现象，再写入 `docs/todo.md` 或拆修复切片。
5. 不要把“没有测”写成“已确认”。这是坏味道，会污染后续所有判断。

## 验收环境

本地源码或 Docker 均可，关键是浏览器访问到项目页面：

```text
http://localhost:3000
```

如果只验证画布、素材和本地存储，不需要外部 AI Key。

## 画布项目导入导出

覆盖：`PT-CANVAS-001`、`PT-CANVAS-002`、`PT-CANVAS-003`。

代码证据入口：

- `web/src/app/(user)/canvas/page.tsx`：画布库页面、导入按钮、多选导出入口。
- `web/src/app/(user)/canvas/components/canvas-project-card.tsx`：单个画布卡片导出入口。
- `web/src/app/(user)/canvas/utils/canvas-export.ts`：生成 zip，写入 `projects.json` 和媒体文件。
- `web/src/app/(user)/canvas/export-types.ts`：导出文件结构。

### 准备数据

1. 打开 `/canvas`。
2. 新建两个画布项目，分别命名为容易识别的名称。
3. 至少在一个项目中加入图片节点或视频节点，保证 zip 里有媒体文件可验证。
4. 回到 `/canvas`。

### 单个画布导出验收

1. 点击某个画布卡片上的导出按钮。
2. 确认下载得到 `.zip` 文件。
3. 解压检查：必须包含 `projects.json`。
4. 如果画布引用了图片或视频，zip 中应包含 `projects/<projectId>/files/` 下的媒体文件。

通过标准：

- `projects.json` 存在。
- `projects.json` 中只有本次导出的项目。
- 媒体节点对应的 `storageKey` 能在 `files` 映射中找到。

### 多画布导出验收

1. 在 `/canvas` 勾选两个画布项目。
2. 点击“导出选中”。
3. 解压 zip，检查 `projects.json` 中包含两个项目。
4. 检查每个项目的媒体文件路径是否位于各自 `projects/<projectId>/files/` 目录下。

通过标准：

- zip 可打开。
- `projects.json` 中项目数量等于勾选数量。
- 每个项目的标题、节点数量、连线数量与导出前一致。

### 导入验收

1. 为避免污染原数据，建议先在另一个浏览器 profile 或清空本地 IndexedDB 后验证。
2. 打开 `/canvas`。
3. 点击“导入画布”，选择刚才导出的 zip。
4. 确认页面提示导入成功，并仍停留在画布库。
5. 打开导入的画布项目。

通过标准：

- 导入后画布项目数量增加。
- 项目标题、节点、连线恢复。
- 图片或视频节点能正常展示，不出现只有节点信息、媒体丢失的状态。

1. 准备一个包含多个节点的画布，其中至少一个图片或视频节点引用不存在的本地 `storageKey`；再准备一条画布助手消息，让其中至少一个图片引用指向不存在的本地 `storageKey`。

## 画布生成运行态并发清理

覆盖：`PT-CANVAS-007`。

### 验收步骤

1. 打开一个含多个可生成节点的画布。
2. 触发第一个节点生成，在它未完成时触发第二个节点生成。
3. 让第一个任务先失败或先完成，同时第二个任务仍在进行。
4. 观察第二个节点的 loading 状态、生成按钮禁用状态和节点状态。
5. 构造一个参考图本地存储已丢失或补水结果为空的节点，发起生成、重试或助手参考图生成。

通过标准：

- 任一任务结束后，只清理它自己的运行态。
- 其它任务仍在进行时，不能被提前清掉 loading 或按钮禁用状态。
- 第二个任务最终完成或失败后，再按自己的结果更新状态。
- 参考图丢失时页面展示可读错误，当前节点进入错误状态，不继续把空参考文件发给上游。

## 图片生成和生图工作台

覆盖：`PT-CANVAS-005`、`PT-IMAGE-001`、`PT-IMAGE-002`、`PT-IMAGE-003`、`PT-IMAGE-004`。

### 验收前提

- 已配置可用模型渠道，或允许浏览器本地直连并配置有效 OpenAI 兼容 API Key。
- 能打开 `/image` 和至少一个 `/canvas/:id` 画布。

### 验收步骤

1. 在 `/image` 调整尺寸、质量、生成张数和模型，确认 UI 状态变化清楚。
2. 发起一次生图请求，确认请求参数符合当前 UI：`auto` 不传 `size`，其它比例或像素尺寸传 `size`。
3. 选择不同质量值，确认 `auto` 或异常值不会直接作为非法 `quality` 发给上游。
4. 生成成功后刷新页面，点击历史记录，确认提示词、参考图、模型、质量、尺寸、张数和结果预览可恢复。
5. 在 `/canvas/:id` 打开生成配置节点的图像参数浮层，修改质量、尺寸和生成张数后发起生成。

通过标准：

- 生图工作台和画布生成配置节点的图像参数行为一致。
- 请求参数和 UI 选择一致。
- 历史记录只依赖 `storageKey` 恢复结果，不长期保存大 base64。
- 上游不支持的质量值不会导致前端侧残留参数直接 400。

## 视频生成和视频创作台

覆盖：`PT-VIDEO-001` 到 `PT-VIDEO-007`。

### 验收前提

- 已配置支持视频生成的模型渠道。
- 能打开 `/video` 和至少一个 `/canvas/:id` 画布。

### 验收步骤

1. 在 `/video` 调整清晰度、尺寸、秒数，确认常用值和手动输入都可用。
2. 选择 `auto` 尺寸发起请求，确认请求不携带 `size`。
3. 输入清晰度数字，确认提交请求时再拼接 `p`。
4. 使用 0、1、多张参考图分别发起视频生成，确认请求使用 `resolution_name`、`preset=normal`、`input_reference[]`。
5. 在画布视频节点附近打开视频设置浮层，确认不被节点悬浮工具栏遮挡，点击面板内容不会误关闭。
6. 触发或模拟视频创建失败、算力点不足或上游失败，确认页面优先展示后端 `{ code, msg }` 里的 `msg`，不继续轮询 `undefined`。
7. 完整跑一次 `/video` 生成，确认结果展示、保存素材、下载和刷新后历史记录回填。
8. 从生图工作台或画布切换到 `/video` 后发起生成，确认请求默认使用视频模型，且没有混入图片生成参数。
9. 给画布视频节点显式指定一个不同的视频模型后发起生成，确认请求使用节点模型，而不是被全局 `videoModel` 覆盖。

通过标准：

- 视频参数面板在画布和视频创作台行为一致。
- 视频请求参数符合目标接口约定。
- 错误处理不产生无效轮询。
- 历史记录能恢复提示词、参考图、参数和媒体结果。

## 后台渠道配置

覆盖：`PT-ADMIN-001`、`PT-ADMIN-002`、`PT-ADMIN-003`、`PT-ADMIN-005`。

### 验收前提

- 已用管理员账号登录。
- 能打开 `/admin/settings`。
- 有至少一个可用于拉取模型列表的渠道配置。

### 验收步骤

1. 新增渠道，打开可用模型弹窗，分别验证“新获取”和“已有”分组。
2. 在弹窗内手动增加模型，再拉取模型列表，确认能写回表单。
3. 编辑已有渠道，API Key 留空或只填空白保存，确认不会触发必填校验，并沿用原密钥。
4. 新增渠道时 API Key 留空保存，确认仍会触发必填校验。
5. 启用、禁用渠道并调整模型，确认公开配置里的系统可用模型候选项由启用渠道模型合并去重生成。
6. 在公开配置中手动勾选最终开放模型，确认候选项和开放项不是一回事。
7. 清空渠道接口地址后拉取模型或测试模型，确认接口返回 HTTP 400，页面展示“缺少接口地址”。
8. 填写格式错误的渠道接口地址后拉取模型或测试模型，确认接口返回 HTTP 400，页面展示“接口地址格式错误”。
9. 填写前后带空白的合法渠道接口地址后拉取模型或测试模型，确认接口仍使用裁剪后的地址请求上游。
10. 编辑已有渠道时把 API Key 填成空白后拉取模型，确认接口沿用已保存密钥请求上游。
11. 对新增渠道清空 API Key 后拉取模型或测试模型，确认接口返回 HTTP 400，页面展示“缺少 API Key”，后端不 panic。
12. 清空测试模型名后测试模型，确认接口返回 HTTP 400，页面展示“缺少模型名称”。
13. 配置错误 API Key、不可达地址，或让渠道上游返回 HTTP 错误后拉取模型或测试模型，确认接口返回 HTTP 502，页面展示可读错误信息。
14. 让渠道上游返回 HTTP 200 但 JSON 业务体为 `{"error":...}`、`{"code":1}`、`{"success":false}` 或 `{"status":"failed"}`，或返回非法 JSON 后拉取模型或测试模型，确认接口返回 HTTP 502，不能显示空模型列表或“ok”。
15. 让测试模型上游返回 HTTP 200 但缺少 `choices`、`choices` 为空或 `choices[0].message.content` 为空，确认接口返回 HTTP 502，不能显示“ok”。

通过标准：

- 新增和编辑渠道的 API Key 校验行为不同。
- 模型弹窗能拉取、手动增加并写回。
- 公开模型候选项只来自启用渠道，且去重。
- 最终开放模型仍由公开配置手动勾选。
- 缺接口地址、接口地址格式错误、API Key 或测试模型名的渠道操作失败清楚，不伪装成 HTTP 200，也不会触发后端崩溃；接口地址前后空白不会导致合法地址误失败，编辑已有渠道时空白 API Key 会沿用已保存密钥。
- 上游 HTTP 错误、HTTP 200 业务错误、非法 JSON 响应、测试模型响应缺少内容或网络失败返回 HTTP 502，且页面仍能展示可读错误。

## 后台用户算力点

- 能打开用户管理页面或直接调用后台用户算力点调整接口。
- 有可控的故障注入环境，用来模拟 `credit_logs` 写入失败。

### 验收步骤

1. 记录目标用户当前算力点余额。
2. 开启 `credit_logs` 写入失败的故障注入。
3. 调整该用户算力点。
4. 确认接口或页面提示失败。
5. 重新读取用户余额和算力流水。

通过标准：

- 调整失败后用户余额保持原值。
- 不产生孤儿 `admin_adjust` 流水。
- 关闭故障注入后，正常调整仍会同时更新余额和写入调整流水。

## API 响应和鉴权

覆盖：`PT-API-001`。

### 验收前提

- 后端服务已启动。
- 能在浏览器开发者工具或接口调试工具里查看 HTTP 状态码和响应体。

### 验收步骤

1. 未登录访问需要用户登录的生成接口，例如 `/api/v1/chat/completions`，确认返回 HTTP 401，响应体仍是 `{ code, data, msg }`。
2. 未登录访问后台接口，例如 `/api/admin/users`，确认返回 HTTP 401，响应体仍是 `{ code, data, msg }`。
3. 使用错误用户名或密码登录，确认返回 HTTP 401，响应体仍是 `{ code, data, msg }`。
4. 注册或后台保存用户时触发用户名为空、包含空格或缺密码，确认返回 HTTP 400。
5. 注册或后台保存用户时触发用户名冲突，确认返回 HTTP 409。
6. 后台编辑一个不存在的用户 ID，确认返回 HTTP 404，且不会创建该 ID 的新用户。
7. 使用后端云端渠道发起 AI 生成，但测试账号算力点不足，确认返回 HTTP 402，响应体仍是 `{ code, data, msg }`。
8. 使用非管理员账号访问后台接口时，确认返回 HTTP 403。
9. 触发 AI 请求缺少 `model` 的场景，确认返回 HTTP 400。
10. 触发 Linux.do OAuth 未开启、缺少客户端配置或上游失败，确认分别返回或传递 403、400、502 语义。
11. 触发退款用户缺失或后台渠道兜底上游错误，确认分别返回或传递 404、502 语义。
12. 触发后端 AI 上游失败时，确认返回 HTTP 502，并且前端仍展示后端 `msg`，不要退化成空泛网络错误。

通过标准：

- 明确鉴权、登录失败、用户校验、Linux.do OAuth 配置错误、退款异常、算力点不足、参数和上游失败不再伪装成 HTTP 200。
- 错误响应结构仍兼容 `{ code, data, msg }`。
- 前端文本 AI 和视频生成请求在非 2xx 时仍优先展示后端 `msg`。

## AI 算力点结算

覆盖：`PT-AI-001`、`PT-AI-002`、`PT-AI-003`、`PT-AI-004`、`PT-AI-005`、`PT-AI-006`、`PT-AI-007`。

### 验收前提

- 后端服务已启动，并使用后端云端渠道发起 `/api/v1/*` 生成请求。
- 能查看后端日志和管理员侧算力流水。
- 测试账号有足够算力点。

### 验收步骤

1. 发起一次会走后端 AI 代理扣费的生成请求。
2. 在上游已返回成功响应后，模拟客户端中断或写出失败。
3. 查看后端日志，确认出现 `AI proxy response copy failed`，并包含已写字节数和错误。
4. 查看对应用户算力流水，确认有本次失败回调产生的返还记录。

通过标准：

- 上游成功但响应写给客户端失败时，不再静默吞掉 `io.Copy` 错误。
- 失败回调会执行退款，用户算力点不应停留在“扣了但没有收到结果”的状态。
- AI POST 上游返回 HTTP 200 但 JSON 业务体包含 `error`、顶层数字或字符串 `code` 非成功值、`status=error/failed/cancelled`、`success=false`，或视频创建响应缺少任务 `id` 时，应按失败返回 HTTP 502，并触发退款；上游漏写或误写成 `text/plain` 但响应体是 JSON 错误时不能绕过业务失败识别，视频创建响应缺失或错误设置 JSON `Content-Type` 时也不能绕过任务 ID 校验；超过 64KB 但未超过后端检查上限的 JSON 失败和视频创建成功响应不能被前缀截断误判；`stream=true` 请求即使请求体很大或上游误标 JSON `Content-Type`，也不应被错误预读卡住，首个 SSE chunk 应及时透传；流式请求返回有明确长度的短 JSON 错误体时，应返回 HTTP 502 并退款，不能当成成功流透传。
- 视频任务创建成功后，消费流水应绑定视频任务 ID；如果绑定任务 ID 失败，本次创建应失败并返还扣点；后续轮询同一任务得到 `failed` / `cancelled` 时，只能退款一次。
- 多个渠道支持同一视频模型时，后续轮询和下载应使用创建任务时绑定的渠道，而不是重新随机按模型选渠道；如果两个渠道共享同一 base URL 但 API Key 不同，应优先用绑定渠道名称精确匹配，不能误用另一个渠道；绑定渠道 base URL 末尾有无 `/v1` 应视为等价；当前用户没有任务绑定流水、已绑定渠道被删除、改名或 URL 无法构造请求时，应返回网关错误，不能随机退回其他同模型渠道；视频模型成本为 0 或未配置成本时，也应保存 0 金额消费流水用于任务渠道绑定。
- 消费扣点和消费流水必须同事务成功；如果 `credit_logs` 写入失败，用户余额不能被扣走。
- 退款返还和退款流水必须同事务成功；如果 `credit_logs` 写入失败，用户余额不能被多加。
- 客户端传入极大 `n` 时，后端不能出现整数溢出、不扣费、负数消费或正数 `ai_consume` 反向流水；超出上限的生成数量应按后端硬上限处理。
- 仍未覆盖的坏味道：更完整的 request/task trace id。

## 部署和数据库迁移

覆盖：`PT-DB-001`、`PT-DOCKER-001`。

### 验收前提

- 使用一份新的 SQLite 数据目录或空数据库。
- 已准备 `.env`，确认 `DB_AUTO_MIGRATE=false`。

### 验收步骤

1. 直接执行 `go run . migrate`，确认命令退出成功。
2. 再执行 `go run .`，确认后端能启动，`GET /api/health` 返回 `ok`。
3. 使用 Docker 镜像或 `docker compose -f docker-compose.local.yml up -d --build` 启动，确认启动脚本会先迁移再启动服务。
4. 检查 SQLite 场景下数据库文件仍落在挂载的 `data` 目录。
5. 在一份新空库上先不执行迁移、直接启动后端，确认服务会失败并提示显式迁移；即使 `ADMIN_USERNAME` / `ADMIN_PASSWORD` 置空禁用默认管理员，也不能绕过这个启动前检查。
6. 用一份表存在但字段不完整的旧 schema 启动后端，确认服务同样失败并提示显式迁移。
7. Docker 启动成功后，在容器内终止内部 Go API 进程，确认容器退出或被 compose 按 `restart` 策略重启，而不是 Next.js 独自继续运行。
8. 再终止 Next.js 进程，确认容器同样退出或重启，内部 Go API 不会孤儿运行。

通过标准：

- 服务启动本身不依赖启动时隐式迁移。
- 未迁移空库或明显过期的 schema 不会被普通服务启动静默放行。
- 显式迁移命令能创建当前应用表。
- Docker 启动路径不要求用户手动进入容器补迁移。
- Docker 运行时不会出现 Go API 或 Next.js 单边退出后的半活容器。
